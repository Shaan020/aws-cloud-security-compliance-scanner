## 📖 Introduction

In cloud environments, one misconfigured setting — a public storage bucket, a firewall rule left open to the internet, or a missing login safeguard — can be the difference between a secure system and a costly breach. This project addresses that gap by automating the audit process itself. It connects directly to a real AWS account using Python and the `boto3` SDK, runs a set of security checks modeled on the CIS AWS Foundations Benchmark, and presents the results through an interactive Streamlit dashboard. Rather than manually clicking through the AWS console to check dozens of settings, this tool consolidates identity, storage, and network security checks into a single automated scan, complete with pass/fail reporting and remediation context for every finding.

This project was built as a self-directed learning initiative to develop hands-on, practical skills at the intersection of **Cloud Security** and **GRC (Governance, Risk, and Compliance)** — bridging technical implementation with the audit and reporting discipline expected in real-world compliance work.

---

## 🎯 What It Does

- Connects to a live AWS account using `boto3`, AWS's official Python SDK
- Runs **5 automated security checks** spanning IAM, S3, and EC2
- Saves every finding to a structured `scan_results.json` file
- Visualizes results on an interactive **Streamlit dashboard** with color-coded pass/fail/warning indicators
- Provides a documented **before-and-after remediation example**, proving the scanner detects real misconfigurations and verifies fixes

---

## 🔍 Security Checks Performed

| # | Check | What It Verifies | Why It Matters |
|:-:|---|---|---|
| 1 | **S3 Public Access** | Whether S3 buckets block public access | Public buckets are one of the most common causes of real-world cloud data breaches |
| 2 | **IAM User MFA** | Whether individual IAM users have MFA enabled | Password-only accounts are vulnerable to credential theft and phishing |
| 3 | **Security Group Exposure** | Whether firewall rules expose risky ports (SSH/22, RDP/3389) to `0.0.0.0/0` | Open remote-access ports are constantly scanned and targeted by automated attacks |
| 4 | **Access Key Age** | Whether IAM access keys are older than 90 days | Long-lived, unrotated credentials increase the risk of undetected compromise |
| 5 | **Root Account MFA** | Whether the AWS root account has MFA enabled | Root has unrestricted access to the entire account with no permission boundaries |

## 🚀 How to Run

```bash
# Install dependencies
pip install boto3 streamlit

# Configure AWS credentials (one-time setup)
aws configure

# Run all security checks
python run_all_checks.py

# Launch the dashboard
streamlit run dashboard.py
```

- The dashboard opens automatically in your browser
- Results are read live from `scan_results.json`, generated by the scan script
- Re-run `run_all_checks.py` anytime to refresh findings after making changes to your AWS account

## 🔧 Demonstrated Fix: Before / After Example

To validate the scanner's accuracy, a test security group (`test-risky-sg`) was deliberately created with port 22 (SSH) open to the entire internet (`0.0.0.0/0`).

**🔴 Before — scanner correctly detects the exposure**
- 7 total checks run, 2 failures identified
- Security group misconfiguration flagged with the message: *"Port 22 open to entire internet"*

![Before Fix](screenshots/before_fix.png)

**🟢 After — issue remediated**
- Security group rule restricted to a specific IP address, removing the `0.0.0.0/0` exposure
- Re-running the scanner confirmed the finding was fully resolved — findings dropped from 7 to 6, with only one remaining unrelated failure

![After Fix](screenshots/after_fix.png)

> This demonstrates the full workflow of a real compliance tool: **detect → remediate → verify.**

## 📝 Notes on Setup

- Checks use IAM permissions scoped to read-only access where possible (`IAMReadOnlyAccess`, `AmazonEC2ReadOnlyAccess`), following the principle of least privilege
- Region is currently hardcoded to a single AWS region; multi-region scanning is a planned extension
- This project focuses on **detection and reporting only** — it does not perform automatic remediation by design; findings are meant to be reviewed before action is taken
- `scan_results.json` is excluded from version control since it may contain real account identifiers — regenerate it locally by running the scanner
